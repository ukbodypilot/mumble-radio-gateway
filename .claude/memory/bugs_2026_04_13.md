## 2026-04-13 — Pi Zero 2W DietPi Rebuild Bugs

### BT firmware missing (BD Address AA:AA:AA:AA:AA:AA)
- DietPi doesn't include `bluez-firmware` package
- BCM43430A1.hcd not found, chip runs with dummy address, invisible to other devices
- Fix: `apt install bluez-firmware`, reboot (hci up/down not enough for UART BT)

### RFCOMM kernel module not loading at boot
- `rfcomm.ko` exists but `modules-load.d` didn't trigger before endpoint started
- Fix: added `ExecStartPre=/sbin/modprobe rfcomm` to `bt-wifi-coex.service`

### bluetoothd --compat needed for SDP
- sdptool requires SDP server, which bluetoothd doesn't start without `--compat` flag
- D75 needs SPP profile registered via `sdptool add SP` + device class `0x1c0104`
- Fix: sed ExecStart in bluetooth.service to add `--compat`

### SCO-over-HCI must be sent before EVERY SCO connect
- Pi Broadcom BCM43430 routes SCO to PCM pins by default
- Vendor command `hcitool cmd 0x3f 0x1c 0x01 0x02 0x00 0x00 0x00` routes to HCI
- Command does NOT persist across BT disconnect/reconnect cycles
- Symptom: SCO socket connects OK but recv() returns 0 bytes
- Fix: send vendor command in AudioManager.connect() before SCO socket creation

### CONFIG_BT_SCO not a separate kernel option
- Investigated "missing CONFIG_BT_SCO" — it's actually compiled into bluetooth.ko via CONFIG_BT_BREDR
- BTPROTO_SCO = 2 (not 6 or 17) — old code had correct value
- SCO sockets work fine once SCO-over-HCI routing is set
